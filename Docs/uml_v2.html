Class diagram | VGP134 Assignment 1 | Final — heterogeneity resolved via std::variant
Resolved since v1: Instructor confirmed Container's MaxSize stays compile-time. Heterogeneity solved via std::variant (A2 flavor — variant of containers, not of items). Each category gets its own compile-time cap. Bag caps locked: Weapons 3 · Medkits 5 · Rations 10. Bag uses InventoryManager (full template machinery for the puzzle layer); stash is a thin Stash struct over std::vector<AnyItem> (uncapped score ledger). No inheritance anywhere.

Type Aliases (locked)

// All three Item instantiations
using WeaponItem = Item<std::string>;   // damage type: "blunt", "piercing", "fire"
using MedkitItem = Item<int>;           // heal amount in HP
using RationItem = Item<float>;         // stamina restore percentage

// The three concrete container types
using WeaponContainer = Container<WeaponItem, 3>;
using MedkitContainer = Container<MedkitItem, 5>;
using RationContainer = Container<RationItem, 10>;

// The variant — one slot, three possible container types
using AnyContainer = std::variant<WeaponContainer, MedkitContainer, RationContainer>;

// For loose items on the ground (single-item variant)
using AnyItem = std::variant<WeaponItem, MedkitItem, RationItem>;

Aliases live in a shared header (e.g. Types.h) — included by InventoryManager, LootOnGround, and Game.

InventoryManager Concrete

Holds AnyContainer per category | iterates via std::visit

Members

  • _categories : std::map<std::string, AnyContainer>

Methods

  • AddCategory(const std::string& name, AnyContainer container)
  • AddItem<ItemT>(category, item) : bool
  • GetTotalValue() const : int
  • GetTotalWeight() const : float
  • FindItemsByValue(int minValue) const : std::vector<AnyItem>
  • SortCategory<Comp>(name, comparator)

Implementation notes

  • Aggregators (GetTotalValue, GetTotalWeight) iterate the map and use std::visit on each AnyContainer to dispatch to the right Container's items
  • AddCategory takes a pre-built variant — caller picks which Container type goes in
  • AddItem is a template: the category name + item type determine which variant alternative to write into. Return false on category miss or container full.
  • Brief's "AddCategory(name, maxSize)" — maxSize is the runtime hint per instructor; here it's encoded in the Container's compile-time N already

enum class GameState Enum

Top-level state machine for the game loop

  • MainMenu  |  Running  |  Paused  |  GameOver

Relationships

  • Container<T,N> ◆→ holds many T internally (composition via std::vector)
  • InventoryManager ◆→ holds many AnyContainer instances (composition via std::map). Each map value is a variant wrapping one of the three concrete Container types.
  • Game ◆→ owns _playerBag (InventoryManager), _stash (Stash), and _loot (vector of LootOnGround)
  • Stash is intentionally light — no Container, no map, no caps. Score readout only.
  • Renderer → reads const Game& at draw time. Game has no reference to Renderer. One-way dependency.
  • std::visit is the dispatcher anywhere we need to "do something" with an AnyContainer or AnyItem. Generic lambdas keep the syntax tolerable.
  • No inheritance anywhere. Heterogeneity is value-based via variant.
